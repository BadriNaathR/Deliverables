User Stories

FR: 10US: 15

As an Agile Coach, I have decomposed the Functional Requirements Document into a structured backlog. The stories are grouped into logical Epics to facilitate sprint planning and architectural layering.



Epic 1: Multi-Channel Ingestion & Data Parsing

Focus: Establishing the pipeline for raw data entry and metadata extraction.


US-001: Ingest Requests via API Endpoints

  • User Story: As a System Integrator, I want to expose secure API endpoints for Email and Chat hooks so that external communications can be captured automatically.
  • Acceptance Criteria:
  • Given a payload from Slack or an Email service.
  • When the endpoint is called with valid JWT credentials.
  • Then the system accepts the raw text and returns a 201 Created status.
  • Story Points: 5 | Priority: High | FR Trace: FR-001, NFR-002

US-002: Automatic Metadata Extraction

  • User Story: As an IT Staff member, I want the system to automatically identify the sender and timestamp of a request so that I don’t have to manually input contact details.
  • Acceptance Criteria:
  • Given an incoming raw message.
  • When the ingestion engine processes the data.
  • Then the Requestor ID, Source Channel, and Timestamp are parsed and stored in the database.
  • Story Points: 3 | Priority: High | FR Trace: FR-002


Epic 2: AI Synthesis & Intelligent Classification

Focus: Leveraging LLMs to transform unstructured text into structured data.


US-003: AI-Generated Note Synthesis

  • User Story: As an IT Staff member, I want the AI to summarize raw requests into a structured format so that I can quickly understand the user's needs without reading long threads.
  • Acceptance Criteria:
  • Given a new raw request.
  • When the AI engine processes the text.
  • Then it must return a JSON object containing a Summary, Context, and Proposed Next Actions within 5 seconds.
  • Story Points: 8 | Priority: High | FR Trace: FR-003

US-004: Automated Categorization and Prioritization

  • User Story: As a Manager, I want the system to suggest a Request Type and Priority level so that the team can focus on critical issues first.
  • Acceptance Criteria:
  • Given the content of a request.
  • When analyzed by the LLM.
  • Then the request is assigned a type (Access, Issue, Info, Change) and a priority (Low to Critical).
  • Story Points: 5 | Priority: Medium | FR Trace: FR-004

US-005: Intelligent Meta-Tagging

  • User Story: As an IT Staff member, I want the system to suggest relevant hashtags (e.g., #Hardware) so that I can easily filter and search for similar historical cases.
  • Acceptance Criteria:
  • Given a synthesized note.
  • When the tagging engine runs.
  • Then at least one relevant meta-tag is suggested based on the request body.
  • Story Points: 3 | Priority: Low | FR Trace: FR-005


Epic 3: Review Interface & Workflow Management

Focus: The human-in-the-loop interface and state transitions.


US-006: Side-by-Side Review Interface

  • User Story: As an IT Staff member, I want to see the raw input and the AI-generated note side-by-side so that I can verify the accuracy of the AI synthesis.
  • Acceptance Criteria:
  • Given a request in Draft state.
  • When I open the Review UI.
  • Then the raw text is displayed on the left and editable AI fields are on the right.
  • Story Points: 5 | Priority: High | FR Trace: FR-006

US-007: Workflow State Enforcement

  • User Story: As a Manager, I want requests to move through a structured lifecycle (Draft -> Reviewed -> Finalized) so that we maintain data integrity.
  • Acceptance Criteria:
  • Given a request in Reviewed state.
  • When I click "Finalize".
  • Then the record status changes to Finalized and all text fields become read-only.
  • Story Points: 3 | Priority: High | FR Trace: FR-007

US-008: Finalization Validation Rules

  • User Story: As an IT Staff member, I want the system to prevent me from finalizing incomplete notes so that our records remain high-quality.
  • Acceptance Criteria:
  • Given a note with a Summary shorter than 20 characters.
  • When I attempt to "Finalize".
  • Then the system displays a validation error and prevents the state change.
  • Story Points: 2 | Priority: Medium | FR Trace: BR-RULE-01


Epic 4: SLA Management & Analytics

Focus: Business logic for deadlines and executive reporting.


US-009: Dynamic SLA Calculation

  • User Story: As a Manager, I want the system to calculate a target resolution date upon finalization so that we can track our performance against service levels.
  • Acceptance Criteria:
  • Given a finalized request with a specific Priority and Type.
  • When the SLA calculator runs.
  • Then it sets a Target Resolution Date excluding weekends and public holidays.
  • Story Points: 5 | Priority: High | FR Trace: FR-008, BR-RULE-02

US-010: Global Request Register with SLA Highlighting

  • User Story: As an IT Staff member, I want a centralized table view of all requests with SLA breach warnings so that I can prioritize my daily workload.
  • Acceptance Criteria:
  • Given the Global Register view.
  • When a request is approaching or past its Target Resolution Date.
  • Then the row is highlighted in red or shows a "Breach" warning.
  • Story Points: 5 | Priority: Medium | FR Trace: FR-009

US-011: Executive Analytics Dashboard

  • User Story: As a Manager, I want to see real-time visualizations of request trends and SLA compliance so that I can allocate resources effectively.
  • Acceptance Criteria:
  • Given the Dashboard view.
  • When data is loaded.
  • Then it displays "Open Requests by Type" and "Average Time to Finalization" charts.
  • Story Points: 8 | Priority: Medium | FR Trace: FR-010


Epic 5: Governance, Security & Audit

Focus: Non-functional requirements and administrative controls.


US-012: SSO & JWT Security Integration

  • User Story: As a Security Admin, I want all users to authenticate via Internal SSO so that we maintain a secure access perimeter.
  • Acceptance Criteria:
  • Given a user attempting to access the UI.
  • When they provide credentials.
  • Then the system validates via SSO and issues a JWT for subsequent API calls.
  • Story Points: 5 | Priority: High | FR Trace: NFR-002

US-013: Comprehensive Audit Logging

  • User Story: As an Admin, I want to see a history of all changes made to a request so that we can perform forensic reviews if data is mishandled.
  • Acceptance Criteria:
  • Given an edit to a request record.
  • When the change is saved.
  • Then a new entry is created in the audit_log with the user ID, timestamp, and a diff of the changes.
  • Story Points: 5 | Priority: High | FR Trace: NFR-003

US-014: Administrative Soft-Delete

  • User Story: As an Admin, I want the ability to delete incorrect records without permanently removing them from the database so that we can recover data if deleted in error.
  • Acceptance Criteria:
  • Given an Admin user.
  • When they select "Delete" on a request.
  • Then the record is flagged as deleted and hidden from the UI, but remains in the database.
  • Story Points: 2 | Priority: Medium | FR Trace: BR-RULE-03

US-015: Containerized Deployment (Scalability)

  • User Story: As a DevOps Engineer, I want the application to be containerized with Docker so that we can scale horizontally during high-traffic periods.
  • Acceptance Criteria:
  • Given a Docker environment.
  • When the application is deployed.
  • Then multiple instances can run using Redis for session management without data loss.
  • Story Points: 5 | Priority: Medium | FR Trace: NFR-004
